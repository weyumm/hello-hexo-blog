# 什么是 GitHub Pages

## 先说结论

GitHub Pages 是 GitHub 提供的一项**静态网页托管服务**。

它能帮你把仓库里的网页文件，直接发布成一个可以访问的网址。  
特别适合拿来放：

- 个人博客
- 学习笔记
- 项目文档
- 作品展示页
- 简单宣传页

如果你已经会写一点 HTML、CSS、JavaScript，那么 GitHub Pages 就像一块“能把网页摆出去”的展示板。

---

## 什么叫“静态网页托管”

这句话里最关键的是两个词：**静态网页** 和 **托管**。

### 静态网页

静态网页的意思是：服务器主要负责把文件原样发给浏览器。  
这些文件通常包括：

- `index.html`
- `style.css`
- `main.js`
- 图片文件

服务器不会在每次访问时临时拼接复杂内容，也不会在服务器端运行你写的数据库逻辑。

你可以先把它理解成：

> 我提前把网页文件准备好，别人访问时，服务器就把这些现成文件发出去。

### 托管

托管就是“别人替你放到网上并提供访问入口”。

如果没有托管服务，你写完网页文件后，它们只躺在你自己的电脑里，别人打不开。  
有了 GitHub Pages，GitHub 会帮你把这些文件部署到网上，让它们拥有一个公开地址。

---

## GitHub Pages 的工作原理

我们用最简单的过程来理解。

### 第一步：你把网页文件放进 GitHub 仓库

比如仓库里有这些文件：

```text
my-site/
├── index.html
├── style.css
└── main.js
```

### 第二步：开启 GitHub Pages

在仓库设置里选择一个发布来源，比如某个分支、某个目录。

### 第三步：GitHub 把这些文件发布成网站

发布后，GitHub 会给你一个网址，例如：

```text
https://你的用户名.github.io/仓库名/
```

当别人访问这个网址时，GitHub 的服务器就会把你仓库中的静态文件返回给浏览器。

### 第四步：浏览器渲染网页

浏览器收到 HTML、CSS、JavaScript 后，就把网页展示出来。

所以整个链路可以概括成：

> 你上传文件 -> GitHub 帮你部署 -> 浏览器读取并显示

---

## 它为什么适合新手

GitHub Pages 对初学者很友好，主要有几个原因。

### 1. 免费

对学习和个人项目来说，通常已经够用。

### 2. 不用自己买服务器

你不用先理解云服务器、Nginx、Linux 运维，先把网页发出去最重要。

### 3. 和 GitHub 仓库天然配合

你每次更新网页文件，本质上就是在更新仓库。  
这让“写代码”和“发布内容”连在了一起。

### 4. 特别适合博客和文档

像 Hexo、Jekyll 这类静态站点工具，都很常见地配合 GitHub Pages 使用。

---

## GitHub Pages 能做什么，不能做什么

这是新手特别容易搞混的地方。

### 能做什么

GitHub Pages 很擅长发布前端静态内容，比如：

- 纯 HTML 网站
- CSS 样式页面
- JavaScript 交互页面
- 静态博客
- 产品介绍页
- 项目说明文档

### 不能直接做什么

它不适合直接承担需要服务器持续计算的后端功能，比如：

- 用户登录系统
- 数据库存储
- 服务端接口逻辑
- 上传文件后在服务器处理

也就是说：

> GitHub Pages 适合展示已经准备好的内容，不适合承担完整后端业务。

如果你的页面要请求数据，也通常是去调用别的 API，而不是指望 GitHub Pages 自己跑后端代码。

---

## 一个现实类比

你可以把 GitHub Pages 想成一个展览橱窗。

- 你先做好海报、卡片、说明牌
- GitHub Pages 负责把这些东西摆出来
- 观众经过时可以看到、点开、阅读

但这个橱窗本身不会替你记账、发货、存会员资料。  
如果你要做这些，就需要更完整的后端服务。

---

## 它和“本地打开 HTML 文件”有什么区别

很多新手第一次写网页时，是双击 `index.html`，直接在浏览器里打开。  
这当然可以用于练习，但它和真正发布到网上还是有区别。

### 本地打开

- 只有你自己的电脑能看
- 地址通常是本地文件路径
- 不方便分享给别人

### 发布到 GitHub Pages

- 别人也能访问
- 有公开网址
- 适合作品展示和长期维护

所以 GitHub Pages 的核心价值，不是帮你“写网页”，而是帮你“发布网页”。

## 和 Hexo 的关系是什么

如果你在用 Hexo，可以把它们理解成“分工合作”的关系：

- Hexo 负责把文章和主题生成静态网页文件
- GitHub Pages 负责把这些静态文件托管到网上

也就是说，Hexo 更像一个“建站工具”，GitHub Pages 更像一个“发布平台”。

这两个经常一起出现，所以很多博客教程都会同时提到它们。

## 一个最小发布思路

假设你现在只有一个最简单的网站：

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8" />
  <title>我的第一个站点</title>
</head>
<body>
  <h1>Hello GitHub Pages</h1>
  <p>这是我发布到网上的第一个页面。</p>
</body>
</html>
```

如果把这个文件作为网站首页发布到 GitHub Pages，那么任何拿到链接的人，都可以访问到它。

这就是“静态网页托管”的最基本形态。

---

## 初学者常见误区

### 1. 以为 GitHub Pages 会帮你写网页

不会。  
它负责托管，不负责替你生成完整内容。

### 2. 以为只要有仓库就自动变成网站

也不会。  
你还需要在仓库设置中开启 GitHub Pages，并选择发布来源。

### 3. 以为所有网站都能丢进 GitHub Pages

不一定。  
如果你的项目必须依赖后端运行，GitHub Pages 就不一定合适。

---

## 你现在应该记住什么

记住这三句话就够了：

1. GitHub Pages 是 GitHub 提供的静态网页托管服务。
2. 它适合发布博客、文档、作品集和简单网页。
3. 它负责“把网页放上网”，不负责“运行复杂后端逻辑”。

## 小结

当你把“网页文件”和“网页托管”这两件事分开理解后，GitHub Pages 就不神秘了。

- HTML/CSS/JS 负责构成网页
- GitHub Pages 负责把网页发布出去

接下来如果你继续学习 Hexo 或其他静态站点工具，就会更容易明白：  
为什么大家总说“写完后部署到 GitHub Pages”。
